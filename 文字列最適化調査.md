# 文字列最適化調査

## 対象
- タイトル画面から「スタート」選択後に遷移する先で、猫イベントの文字列変数表示が遅い。

## 調査結果
- タイトル用イベントはスタート選択後に `movePosition(mapId: 1, x: 12, y: 8)` を実行する。
- 遷移先は `map 1`。
- `map 1` 上で猫チップ `CharaChip/[Animal]Cat_pochi.png` を使うイベントは以下の 5 件。

| event id | name | note |
| --- | --- | --- |
| 18 | 宿のニワトリ | 通常会話 |
| 19 | 分岐のデバッグ用 | 分岐確認 |
| 21 | ループのデバッグ用 | ループ確認 |
| 22 | ループのデバッグ用 | 文字列ピクチャ確認 |
| 23 | 文字列操作のデバッグ用 | **今回の対象** |

## 特定した遅延イベント
- 対象イベント: `map 1 / event id 23 / 文字列操作のデバッグ用`
- コマンド構成:
  1. `message`: `文字列操作のデバッグです`
  2. `callEvent`: `rawEventId: 500215`
  3. `unknown`

## 実質的なホットパス
- `event 23` 自体は薄く、実際の文字列処理本体は呼び出し先の共通イベント `500215`。
- `common event 215` の名称は `セルフ変数変化を確認`。
- この共通イベントは `\cself[5]` を何度も描画するため、`cself` の解決コストがそのまま体感遅延につながる。
- `Changeable DB[0:0:0]` の field 0 (`主人公ステータス / キャラ名`) は文字列カラムで、実値は **`ヒーローさん`**。
- したがって、このイベントが本来表示したい文字列変数は `\cself[5] = ヒーローさん`。

## 実装方針
- `interpolateString()` での `cself` 解決を軽量化する。
- 共通イベント参照を 1 回で済ませ、同一補間内での繰り返し探索を避ける。
- 文字列更新後に新しい値が即時反映されることは維持する。

## 追加調査: 9 回 A ボタンが必要な理由
- これは入力受付が 9 回あるのではなく、`common event 215` 側に **9 個の `message` コマンド** が並んでいるため。
- 内訳は `数値変数は1` が 1 回、`文字列変数はヒーローさん` が 8 回。
- そのため、素直に 1 メッセージ 1 決定入力で進める実装だと、次のステップまで 9 回の決定入力が必要になる。
- これは入力処理の問題ではなく、**common event 215 のコマンド解釈がまだ不十分な可能性**が高い。
